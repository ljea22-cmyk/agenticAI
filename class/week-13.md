## 12.1 배포 옵션과 고려사항

- 에이전트 시스템의 배포 방식 결정 요소
  - 팀 규모
  - 예상 트래픽
  - 예산

### 로컬 실행

- 개발과 테스트 환경에서 가장 간단한 방법
- 장점
  - 빠른 피드백 가능
  - 디버깅이 용이
- ⚠ 단점
  - 확장성 없음 (트래픽 증가에 대응 불가)
  - 단일 장애점 (서버 하나가 죽으면 서비스 전체 중단)

### Docker 컨테이너

- 이식성(Portability)과 재현성(Reproducibility)을 제공하는 컨테이너 기술
- 동일한 이미지를 개발·스테이징·프로덕션 환경에서 모두 실행 가능
  - → 환경 차이로 인한 "내 컴퓨터에서는 됐는데..." 문제 해소
- 소규모 프로덕션에 적합
- Kubernetes 등 오케스트레이션 도구(여러 컨테이너를 자동으로 관리하는 도구)와 결합하면 확장도 가능

### 클라우드 배포

- 대규모 트래픽과 고가용성(High Availability, 장애 없이 지속 운영)이 필요한 경우 적합
- 주요 제공 서비스
  - AWS, GCP, Azure 등
  - 서버리스 함수 (코드만 올리면 서버 관리 없이 실행)
  - 컨테이너 서비스
  - 관리형 Kubernetes
- ⚠ 단점
  - 복잡성 증가
  - 비용 증가

**표 12.1** 배포 옵션 비교

| 옵션 | 장점 | 단점 | 적합한 상황 |
|:---|:---|:---|:---|
| 로컬 | 간단, 빠른 피드백 | 확장 불가, 단일 장애점 | 개발, 테스트 |
| Docker | 이식성, 재현성 | 오케스트레이션 필요 | 소규모 프로덕션 |
| 클라우드 | 확장성, 고가용성 | 복잡성, 비용 | 대규모 프로덕션 |

## 12.2 관측 가능성 (Observability)

- 관측 가능성(Observability): 시스템의 내부 상태를 외부에서 파악할 수 있는 능력
- LLM 애플리케이션은 비결정적이므로
  - 전통적인 모니터링(서버 CPU/메모리 확인 수준)만으로는 부족
  - 요청 내용, 모델 응답, 실패 원인 등을 더 세밀하게 추적해야 함

### 관측 가능성의 세 요소

- **로깅(Logging)**: 이벤트를 시간순으로 기록
  - 포함 내용: 요청 내용, 응답 내용, 에러 메시지 등
- **트레이싱(Tracing)**: 요청의 전체 흐름을 추적
  - 에이전트가 여러 단계를 거칠 때
    - 각 단계의 입력과 출력을 기록
    - 각 단계의 소요 시간을 기록
- **메트릭(Metrics)**: 수치화된 측정값
  - 요청 수
  - 응답 시간
  - 에러율
  - 토큰 사용량
  - → 이들을 시계열(시간 흐름에 따라)로 수집

### 주요 도구

- **Langfuse**
  - 가장 많이 사용되는 오픈소스 LLM 관측 도구
  - 제공 기능: 포괄적 트레이싱, 평가, 프롬프트 관리, 메트릭
  - MIT 라이선스로 셀프호스팅(자체 서버에 직접 설치) 가능
  - OpenTelemetry 및 LangChain에 통합됨

- **LangSmith**
  - LangChain 팀이 만든 관리형 관측 스위트(서비스 형태)
  - LangChain/LangGraph 생태계와 심층 통합
  - 제공 기능: 대시보드, 알림, HITL(사람이 직접 평가) 평가
  - ⚠ 셀프호스팅은 Enterprise 플랜에서만 가능

```python
from langfuse.decorators import observe

@observe()
def process_query(query: str) -> str:
    # 자동으로 트레이스가 기록됨
    return llm.invoke(query)
```

## 12.3 운영 메트릭 설계

- 효과적인 모니터링을 위해 추적할 메트릭(측정 지표)을 미리 정의해야 함

### 핵심 메트릭

- **요청 수**: 시스템 사용량을 나타냄
- **응답 시간**: 사용자 경험을 결정하는 핵심 지표
- **토큰 사용량**: 비용과 직결됨
- **에러율**: 시스템 안정성을 보여줌
- **성공률**: 작업 완료 비율 (에이전트가 목표를 달성한 비율)

### SLI와 SLO

- **SLI (Service Level Indicator, 서비스 수준 지표)**
  - 측정 가능한 메트릭
  - → 예시: "p95 응답 시간" (상위 95%의 요청이 응답되는 시간), "성공률"
- **SLO (Service Level Objective, 서비스 수준 목표)**
  - SLI에 대해 달성하려는 목표값
  - → 예시: "p95 응답 시간 < 5초", "성공률 > 99%"

### 알림 설정

- 임계값(특정 수치 기준)을 기반으로 알림을 설정
  - 에러율이 5%를 초과하면 → 즉시 알림 발송
  - 응답 시간이 10초를 초과하면 → 경고 알림 발송
  - 일일 비용이 예산의 80%에 도달하면 → 주의 알림 발송

## 12.4 비용 추정과 최적화

- LLM API는 토큰(텍스트를 쪼갠 단위) 기준으로 과금
  - → 비용 관리가 운영의 핵심 과제

### 토큰 기반 과금 구조

- 대부분의 LLM API는 입력 토큰과 출력 토큰을 별도로 과금
- ⚠ 출력 토큰이 입력 토큰보다 비쌈
  - → 응답 길이 제어가 비용 절감에 효과적

**표 12.2** 주요 LLM API 비용 (2026년 기준, USD/1M 토큰)

| 모델 | 입력 | 출력 | 특징 |
|:---|---:|---:|:---|
| Claude Opus 4.6 | $5.00 | $25.00 | 최고 성능, 적응형 사고 |
| Claude Sonnet 4.5 | $3.00 | $15.00 | 균형 잡힌 성능·비용 |
| Claude Haiku 4.5 | $1.00 | $5.00 | 빠른 응답, 비용 효율 |
| GPT-4o | $2.50 | $10.00 | 고성능 멀티모달 |
| GPT-4o-mini | $0.15 | $0.60 | 최저 비용 |

### 비용 추정 공식

```
월간 비용 = (입력 토큰 × 입력 단가 + 출력 토큰 × 출력 단가) × 30일
```

- → 예시: GPT-4o-mini로 일일 10만 입력 토큰, 5만 출력 토큰 사용 시
  - 일일 비용: 약 $0.045
  - 월간 비용: 약 $1.35

### 최적화 전략

- **① 모델 선택 최적화**
  - 모든 작업에 최고 성능 모델(GPT-4o 등)이 필요한 것은 아님
  - 단순 분류, 요약, 대화 → GPT-4o-mini나 Claude Haiku로 충분할 수 있음
  - 복잡한 추론에만 → 고성능 모델 사용

- **② 프롬프트 최적화**
  - 불필요한 지시문과 예시를 제거하여 토큰 절약
  - → 예시: 시스템 프롬프트 압축만으로 62%의 크기 감소 보고됨

- **③ 캐싱 활용**
  - 동일한 프롬프트에 대한 응답을 캐싱하면 API 호출 횟수 절감
  - Anthropic의 프롬프트 캐싱: 캐시 읽기가 기본 비용의 단 10%

- **④ 응답 길이 제어**
  - `max_tokens`를 적절히 설정하여 불필요하게 긴 응답 방지

## 12.5 보안과 키 관리

- API 키: 외부 서비스에 대한 접근 권한을 부여하는 민감한 정보
  - → 노출 시 무단 사용, 비용 폭증, 데이터 유출 위험

### API 키 관리 원칙

- ⚠ API 키를 코드에 절대 하드코딩하지 않음
- 올바른 방법
  - 환경 변수 사용
  - `.env` 파일 사용 (git에 업로드 금지)
  - 시크릿 매니저 사용 (AWS Secrets Manager, HashiCorp Vault 등)

```python
import os
from dotenv import load_dotenv

load_dotenv()
api_key = os.getenv("OPENAI_API_KEY")
```

### 키 로테이션 (Key Rotation, 주기적인 키 교체)

- 정기적으로 API 키를 교체
- 즉시 교체해야 하는 경우
  - 키가 외부에 노출되었을 때
  - 해당 키를 알고 있는 직원이 퇴사했을 때
- 교체 절차 (서비스 중단 없이 진행)
  1. 새 키 발급
  2. 애플리케이션에 새 키 적용 및 업데이트
  3. 구 키 폐기

### 최소 권한 원칙 (Principle of Least Privilege)

- 작업에 필요한 권한만 부여
  - → 예시: 읽기 전용 작업 → 읽기 권한만 가진 키 발급
- 프로덕션 환경과 개발 환경의 키를 반드시 분리

### 접근 제어와 감사 (Audit)

- 누가 키에 접근했는지 기록
- 키 사용 내역을 정기적으로 감사(점검)
- 비정상적인 사용 패턴 탐지 시 즉시 알림 발송

## 12.6 프로덕션 체크리스트

- 프로덕션(실서비스) 배포 전 반드시 확인해야 할 항목들

### 배포 전 체크리스트

- 환경 변수와 시크릿이 올바르게 설정되었는지 확인
- API 키가 코드에 하드코딩되지 않았는지 검증
- 의존성 버전이 고정(pin)되어 있는지 확인 (버전 변경으로 인한 호환성 문제 방지)
- 에러 핸들링과 재시도 로직이 구현되어 있는지 점검

### 모니터링 체크리스트

- 요청/응답 로깅이 활성화되어 있는지 확인
- 민감 정보(개인정보, 키 등)가 로그에서 마스킹(가려짐)되는지 검증
- 핵심 메트릭이 추적되는지 확인
  - 요청 수
  - 응답 시간
  - 에러율
  - 비용
- 알림 임계값이 설정되어 있는지 점검

### 보안 체크리스트

- API 키가 시크릿 매니저에 저장되어 있는지 확인
- 키 로테이션 절차가 수립되어 있는지 검증
- 최소 권한 원칙이 적용되어 있는지 점검
- 감사 로그(누가 언제 무엇을 했는지 기록)가 활성화되어 있는지 확인

### 비용 관리 체크리스트

- 예상 월간 비용이 계산되어 있는지 확인
- 비용 한도(예산 상한선)가 설정되어 있는지 검증
- 비용 이상 알림이 설정되어 있는지 점검

## 12.7 관측성 플랫폼 비교

- 12.2절에서 학습한 관측 가능성의 세 요소(로깅·트레이싱·메트릭)를 기반으로
- 2026년 기준 LLM 관측성 플랫폼 5종을 비교
- 목적: 프로젝트에 적합한 도구 선택 기준 제시

### 5대 관측성 플랫폼

**표 12.3** 관측성 플랫폼 비교 (2026년 기준)

| 플랫폼 | 배포 모델 | 무료 티어 | 핵심 강점 | 최적 사용 시나리오 |
|:---|:---|:---|:---|:---|
| LangSmith | 관리형 SaaS | 5,000 트레이스/월 | LangChain/LangGraph 네이티브 통합 | LangChain 기반 프로젝트 |
| Langfuse | 오픈소스 + 클라우드 | 50,000 관찰/월 | 셀프호스팅, MIT 라이선스 | 데이터 주권이 중요한 조직 |
| Arize Phoenix | 오픈소스 + 엔터프라이즈 | 오픈소스 무제한 | 실험 추적, 드리프트 탐지 | ML 팀의 실험 관리 |
| Braintrust | 관리형 SaaS | 무료 시작 | 평가 중심 워크플로우 | 프롬프트 최적화·A/B 테스트 |
| Helicone | 관리형 SaaS + 오픈소스 | 10,000 요청/월 | 비용 추적 특화, 프록시 방식 | 비용 모니터링이 핵심인 팀 |

### 선택 기준

- 플랫폼 선택은 세 가지 축으로 판단

- **① 프레임워크 호환성**
  - LangChain/LangGraph 사용 시 → LangSmith가 가장 긴밀하게 통합
  - 특정 프레임워크에 구애받지 않는 경우 → Langfuse나 Arize Phoenix가 유연

- **② 배포 모델**
  - 데이터를 외부 서버에 보낼 수 없는 규제 환경(금융, 의료 등)
    - → Langfuse(셀프호스팅) 또는 Arize Phoenix(오픈소스) 적합
  - 빠른 시작이 목표인 경우
    - → 관리형 SaaS (LangSmith, Braintrust, Helicone) 편리

- **③ 핵심 관심사**
  - 비용 추적이 최우선 → Helicone
  - 평가 워크플로우가 중요 → Braintrust
  - 포괄적 관측이 필요 → LangSmith 또는 Langfuse

## 12.8 비용 최적화: 설계 시점부터

- 12.4절의 기본 최적화 전략에서 한 단계 더 나아가
- 설계 단계에서 적용할 수 있는 고급 비용 최적화 기법을 다룸

### 모델 라우팅 (Model Routing)

- 모든 작업에 동일한 모델을 사용하는 것은 비용 낭비
- 작업 복잡도에 따라 적절한 모델을 자동 선택하는 라우팅(경로 배정) 전략 필요

**표 12.4** 작업 복잡도별 모델 라우팅 전략

| 작업 유형 | 권장 모델 | 입력 단가 | 비용 절감률 |
|:---|:---|---:|:---:|
| 단순 분류·추출 | Claude Haiku 3 | $0.25/MTok | 기준 |
| 요약·대화·검색 | Claude Haiku 4.5 | $1.00/MTok | — |
| 복잡한 추론·코딩 | Claude Sonnet 4.5 | $3.00/MTok | — |
| 최고 품질 필요 | Claude Opus 4.6 | $5.00/MTok | — |

- 라우팅 로직 구현 방법
  - 규칙 기반: 키워드 매칭, 토큰 수 기준
  - 경량 분류기(작은 ML 모델)로 복잡도 판단
- ⚠ 주의: 라우팅 자체의 비용이 절감 효과보다 작아야 의미 있음

```python
def route_model(query: str, complexity: str) -> str:
    if complexity == "simple":
        return "claude-haiku-4-5-20251001"
    elif complexity == "complex":
        return "claude-sonnet-4-5-20250929"
    return "claude-opus-4-6"
```

### 적응형 추론 제어 (Effort Level)

- Anthropic API의 `output_config.effort` 파라미터로 모델의 추론 깊이를 제어
- 4단계 옵션
  - `low`: 빠르고 간단한 추론 (토큰 사용량 최소)
  - `medium`: 중간 수준 추론
  - `high`: 기본값, 충분한 추론
  - `max`: Opus 4.6 전용, 최대 추론 깊이
- 낮은 effort 사용 효과
  - 토큰 사용량 감소
  - 응답 시간 단축
- 적용 가이드
  - 단순 분류 작업 → `low` 사용 시 `high` 대비 토큰 크게 감소
  - 복잡한 에이전트 코딩, 심층 분석 → `high` 또는 `max` 적합

### 프롬프트 캐싱

- 반복되는 시스템 프롬프트나 컨텍스트(맥락 정보)를 캐싱(미리 저장)하여 비용 절감
- Anthropic 프롬프트 캐싱 특징
  - 캐시 읽기 비용: 기본 입력 비용의 단 10%
  - 캐시 유효 기간 선택 가능
    - 5분 캐시 (기본)
    - 1시간 캐시 (쓰기 비용은 기본 입력의 2배이나, 반복 사용 시 절약)
- 에이전트 시스템에서 특히 효과적인 이유
  - 시스템 프롬프트와 도구 정의가 매 요청마다 반복됨
  - → 예시: 도구 정의가 10,000 토큰이고 분당 10회 호출 시, 캐싱으로 매달 수백 달러 절약 가능

### 배치 API 활용

- 실시간 응답이 필요 없는 작업에 배치(일괄 처리) API 사용
  - 적합한 작업: 대량 분류, 일괄 평가, 로그 분석
- 배치 API 혜택
  - 입력 토큰 50% 할인
  - 출력 토큰 50% 할인
- 처리 보장: 24시간 내 완료
- → 비동기(즉각 결과 불필요) 파이프라인에 적합

## 12.9 MCP Registry와 서버 발견

- 3~4장에서 MCP 개념과 서버 구현을 학습
- 프로덕션 환경에서는 수십 개의 MCP 서버를 관리해야 함
  - → 에이전트가 필요한 서버를 자동으로 발견하는 메커니즘이 필요

### MCP Registry의 개념

- MCP Registry: 사용 가능한 MCP 서버의 정보를 중앙에서 관리하는 디렉터리 서비스
- Registry가 관리하는 메타데이터(서버 설명 정보)
  - 서버 이름
  - 서버 설명
  - 엔드포인트 (서버에 접속하는 주소)
  - 인증 방식
  - 도구 목록
- 활용 방식
  - 에이전트가 Registry에 질의(검색)
  - 작업에 필요한 서버를 동적으로(자동으로) 발견
  - 해당 서버에 연결

### MCP 마켓플레이스

- 커뮤니티 기반 MCP 서버 배포 플랫폼이 등장 중
- **Smithery**
  - MCP 서버를 검색하고 원클릭으로 설치할 수 있는 레지스트리
- **mcp.run**
  - 샌드박스(격리된 안전한 환경)에서 MCP 서버를 안전하게 실행하는 서비스
- 이러한 플랫폼의 역할
  - 서버 발견 표준화
  - 버전 관리 표준화
  - 보안 검증 표준화

### 프로덕션 MCP 서버 관리

- 프로덕션 환경에서 MCP 서버를 운영할 때 고려할 사항

- **① 서버 등록과 발견**
  - 각 서버는 자신의 역할(capabilities, 제공 가능한 기능)을 Registry에 등록
  - 에이전트는 작업 유형에 따라 적절한 서버를 질의하고 연결

- **② 버전 관리**
  - MCP 서버의 도구 스키마(도구의 입출력 형식 정의)가 변경되면 에이전트와의 호환성이 깨질 수 있음
  - → 시맨틱 버저닝(Semantic Versioning, 버전 번호 체계) 적용 필요
  - → 하위 호환성 유지 정책 수립 필요

- **③ 모니터링**
  - 각 MCP 서버의 상태를 지속 추적
    - 가용성 (서버가 살아 있는지)
    - 응답 시간
    - 에러율
  - 서버 장애 시 폴백(fallback, 대안 서버나 방법으로 전환) 전략 마련

- **④ 보안**
  - MCP 서버는 외부 시스템에 대한 접근 권한을 가짐
  - → OAuth 2.1 기반 인증 적용 (표준 인증 프로토콜)
  - → 최소 권한 원칙 적용
  - 4장에서 학습한 보안 패턴이 그대로 적용됨

## 12.11 실패 사례와 교훈

- 프로덕션 운영 시 실제로 발생할 수 있는 문제와 대응 방법

### 예상치 못한 비용 폭증

- 원인
  - 새로운 기능 출시 후 트래픽 급증
  - 무한 루프로 API를 반복 호출하는 버그 발생
- 대응 방법
  - 일일/월간 비용 한도 사전 설정
  - 서킷 브레이커(Circuit Breaker, 임계값 초과 시 자동으로 요청 차단하는 패턴) 구현
    - 한도 도달 시 자동으로 추가 요청 차단

### API 키 노출 사고

- 원인
  - GitHub 등 공개 저장소에 API 키가 포함된 코드를 실수로 업로드
  - → 수분 내에 악의적 행위자가 키를 발견하고 악용
- 예방 방법
  - git-secrets 또는 pre-commit hook 사용
    - 커밋 전 키 포함 여부 자동 검사
- 사고 발생 시 대응
  - 노출된 키 즉시 폐기
  - 새 키 즉시 발급 및 적용

### 모니터링 누락으로 인한 장애 지연 탐지

- 문제
  - 모니터링이 없으면 장애를 사용자 불만 신고로 뒤늦게 알게 됨
- 대응 방법
  - 핵심 메트릭에 대한 알림을 반드시 설정
  - 정기적으로 알림이 실제로 작동하는지 테스트

### 용량 초과 문제 (Rate Limit)

- 원인
  - API 레이트 리밋(일정 시간 내 최대 요청 수 제한)에 도달하여 요청 거부
- 대응 방법
  - 해당 API의 레이트 리밋 수치를 사전에 파악
  - 요청을 시간적으로 분산 처리
  - 지수 백오프(Exponential Backoff, 재시도 간격을 점점 늘리는 전략)로 재시도
  - 필요 시 API 제공사에 레이트 리밋 증가 요청
