## 개요

- 대형 언어 모델(LLM)은 유창한 텍스트를 생성하지만, 때로는 사실과 다른 정보를 자신감 있게 제시함
  - 이러한 현상을 **환각(hallucination)** 이라 하며 실무 시스템에서 심각한 문제를 야기할 수 있음

  - 모델이 그 문서에 없는 내용을 추가하거나 왜곡할 가능성은 여전히 존재함
- 이 장에서 다루는 내용

## 10.1 LLM 환각의 이해

- **환각(hallucination)** 이란
  - 모델이 학습 데이터에 근거하지 않거나
  - 주어진 컨텍스트와 일치하지 않는 정보를 생성하는 현상
- 2025년 OpenAI 연구 결과
  - 환각은 단순한 버그가 아니라 **구조적 특성**임
  - 원인: next-token 예측(다음 토큰을 예측하는 방식)이라는 학습 목표에서 비롯됨
  - 모델은 정확한 답변보다 "자신감 있어 보이는" 답변에 더 높은 보상을 받도록 학습됨

### 환각의 유형

- **첫째: 사실 오류(Factual Hallucination)**
  - 실제와 다른 정보를 생성하는 것
  - → 예시: "파이썬은 1995년에 출시되었다"라고 답변 (실제 출시 연도는 1991년)

- **둘째: 논리 오류(Logical Hallucination)**
  - 주어진 전제로부터 도출될 수 없는 결론을 제시하는 것
  - → 예시: RAG 시스템에서 검색된 문서에 "A는 B보다 빠르다"라고 되어 있는데, "따라서 B가 더 효율적이다"라고 결론짓는 경우

- **셋째: 출처 조작(Source Fabrication)**
  - 존재하지 않는 참고문헌을 인용하거나
  - 실제 논문의 내용을 왜곡하여 인용하는 것
  - ⚠ 학술 정보 시스템에서 특히 위험한 유형

### 환각 발생 원인

- **학습 데이터의 편향**
  - 모델이 특정 패턴을 과도하게 학습할 수 있음
- **불충분한 컨텍스트**
  - 주어진 컨텍스트가 부족할 때 모델은 내부 지식으로 공백을 채우려 함
  - 이 과정에서 오류가 발생함
- **토큰 확률 분포 기반 생성 방식**
  - 생성 방식 자체가 때로는 사실보다 유창함을 우선시하게 만듦
- **Semantic Entropy 연구** (Nature 게재)
  - 모델이 동일한 질문에 대해 의미적으로 다른 답변을 생성할 때 높은 불확실성을 보인다는 점을 활용
  - 환각 탐지 방법 제안
  - 특징: 단어 수준이 아닌 **의미 수준**에서 불확실성을 측정 → 획기적

## 10.2 교차검증 전략

- **교차검증(Cross-Verification)** 이란
  - 동일한 질문에 대해 여러 경로로 답변을 생성하고, 그 일관성을 확인하는 방법
  - 일관성이 낮다면 모델이 확신 없이 답변하고 있을 가능성이 높음

### Self-Consistency

- Wang et al.(2023)이 제안한 방법
- 동작 원리
  - 동일한 모델에게 같은 질문을 여러 번 하되
  - 높은 temperature(예: 0.8~1.2)를 사용하여 다양한 답변을 유도
  - 이후 다수결(majority voting)로 최종 답변을 선택
- 핵심 아이디어
  - 모델이 확신하는 답변은 여러 번 물어도 일관되게 나타남
  - 불확실한 영역에서는 샘플링마다 다른 답변이 나옴

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.8)
samples = [llm.invoke(prompt) for _ in range(3)]
```

### Chain-of-Verification (CoVe)

- Meta AI 연구팀이 개발한 방법
- 4단계 프로세스
  1. 초기 답변을 생성
  2. 그 답변을 검증하기 위한 질문을 계획
  3. 각 검증 질문에 독립적으로 답변
  4. 검증 결과를 바탕으로 최종 답변을 수정
- 실험 결과
  - CoVe를 적용한 LLaMA-65B 모델: FactScore가 63.7 → 71.4로 향상
  - 기술 문서와 코드 리뷰에서: 40%의 정확도 향상이 보고됨

### 전략 비교

**표 10.1** 교차검증 전략 비교

| 전략 | 방법 | API 호출 | 신뢰도 향상 |
|:---|:---|:---:|:---:|
| Self-Consistency | 동일 모델 N회 샘플링 | N회 | 중간 |
| Multi-Model | 서로 다른 모델로 검증 | 모델 수만큼 | 높음 |
| Chain-of-Verification | 검증 전용 프롬프트 추가 | 3~4회 | 중간~높음 |

## 10.3 출처 일치 검증

- RAG 시스템에서 특히 중요한 사항
  - 생성된 답변이 실제로 검색된 문서에 근거하는지 확인하는 것
  - 이를 **Grounding** 또는 **Faithfulness 검증**이라고 함

### Grounded Generation

- **Grounded Generation** 이란
  - 모델이 답변을 생성할 때 반드시 주어진 컨텍스트에서 근거를 찾도록 강제하는 접근
  - 기본: 프롬프트에 "컨텍스트에 없는 정보는 추측하지 마세요"와 같은 지시를 포함
- ⚠ 이러한 지시만으로는 충분하지 않음
  - 모델은 여전히 내부 지식을 활용하여 컨텍스트를 "보완"하려 할 수 있음
  - 따라서 **생성 후 검증**이 반드시 필요함

### NLI 기반 검증

- **NLI(Natural Language Inference, 자연어 추론)** 모델 활용
  - "답변이 컨텍스트로부터 논리적으로 도출되는가"를 자동으로 평가할 수 있음
- **SummaC** 방법
  - 문서-요약 쌍에서 문장 수준 NLI 점수를 집계하는 방법
  - RAG 답변 검증에도 적용 가능

### Faithfulness vs Correctness

- ⚠ **Correctness(정확성)** 와 **Faithfulness(충실성)** 는 다름
  - 답변이 사실적으로 맞더라도(Correct)
  - 그 정보가 주어진 컨텍스트에서 온 것이 아니라면 Unfaithful(불충실)한 것임
- RAG 시스템의 신뢰성은
  - 모델이 실제로 검색된 문서를 사용했는지에 달려 있음
- ⚠ **Post-hoc citation(사후 인용) 문제**
  - 일부 RAG 최적화 모델은 내부 지식으로 답변을 생성한 후 토큰 유사성을 기반으로 사후적으로 인용을 붙이는 경향이 있음
  - 이 문제를 탐지하려면 생성 과정 자체를 검증해야 함

## 10.4 품질 게이트 설계

- **품질 게이트(Quality Gate)** 란
  - 생성 파이프라인의 특정 지점에 배치되어 출력이 기준을 충족하는지 검사하는 검문소
  - 게이트를 통과하지 못하면 재시도하거나 에스컬레이션함

### 게이트 배치 위치

- 효과적인 품질 게이트는 다음 세 지점에 배치

**그림 10.1** 품질 게이트가 포함된 생성 파이프라인

```
[입력] → [생성] → [Gate 1: 형식] → [Gate 2: 출처] → [Gate 3: 일관성] → [출력]
              ↓ 실패        ↓ 실패          ↓ 실패
           [재시도]      [재시도]       [사람 승인]
```

- **Gate 1: 형식 검사**
  - JSON 파싱 가능 여부 확인
  - 필수 필드 존재 여부 확인
- **Gate 2: 출처 일치 검증**
  - 답변이 컨텍스트에 근거하는지 평가
- **Gate 3: 일관성 검사**
  - Self-Consistency를 통해 답변의 안정성 확인

### 게이트 통과 기준

- 각 게이트는 수치적 임계값을 기준으로 통과 여부를 결정
  - 일관성 점수: **0.25 이상**
  - 출처 점수: **0.7 이상**
  - ⚠ 이 값은 도메인과 요구 사항에 따라 조정해야 함
- ⚠ 임계값 설정 시 주의
  - 임계값이 너무 높으면: 정상적인 답변도 거부하는 **False Positive** 증가
  - 임계값이 너무 낮으면: 문제 있는 답변이 통과하는 **False Negative** 증가
  - 실제 운영에서는 로그를 분석하여 최적의 균형점을 찾아야 함

### 실패 시 처리 전략

- **재시도 (기본 전략)**
  - 동일한 요청을 다시 처리하면 LLM의 비결정적 특성으로 인해 다른 결과가 나올 수 있음
  - ⚠ 무한 재시도는 비효율적이므로 최대 재시도 횟수(예: 3회)를 설정
- **최대 재시도 초과 시**
  - 사람에게 에스컬레이션(상위 단계로 이관)하거나
  - 답변 불가 메시지를 반환
  - 어떤 전략을 선택할지는 시스템의 요구 사항에 따라 달라짐

## 10.5 검증 파이프라인 구현

- **LangGraph** 를 활용하면 검증 로직을 워크플로우로 구조화할 수 있음
  - 각 검증 단계를 노드(node)로 정의
  - 조건부 엣지(conditional edge)로 재시도와 에스컬레이션 흐름을 구현

### 상태 정의

- 검증 워크플로우의 상태(state)에 포함되는 정보
  - 질문(query), 컨텍스트(context), 답변(answer)
  - 검증 점수(consistency_score, grounding_score)
  - 재시도 횟수(retry_count)
  - 사람 승인 여부(needs_human_approval)

```python
class VerificationState(TypedDict):
    query: str
    context: str
    answer: str
    consistency_score: float
    grounding_score: float
    retry_count: int
    needs_human_approval: bool
```

### 검증 노드 설계

- 각 검증 단계는 독립된 노드로 구현
  - **consistency 노드**: Self-Consistency를 수행하여 일관성 검사
  - **grounding 노드**: LLM을 활용하여 답변이 컨텍스트에 근거하는지 Grounding 점수 계산
  - **gate 노드**: 모든 검증 결과를 종합하여 다음 단계를 결정

```python
graph.add_node("consistency", check_consistency)
graph.add_node("grounding", check_grounding)
graph.add_node("gate", quality_gate)
```

### 조건부 라우팅

- 품질 게이트 노드는 세 가지 경로 중 하나를 선택
  - 모든 검증을 통과하면 → **종료(END)**
  - 재시도 횟수가 남아 있으면 → **생성 노드로 복귀**
  - 재시도를 초과하면 → **사람 승인 노드로 이동**

```python
graph.add_conditional_edges(
    "gate",
    route_after_gate,
    {"generate": "generate", "human": "human", "end": END}
)
```

## 10.6 OWASP AI Agent Security

- 에이전트 시스템은 전통적인 LLM 애플리케이션과 다른 **고유한 보안 위협**에 노출됨
- **OWASP(Open Worldwide Application Security Project, 오픈 웹 애플리케이션 보안 프로젝트)** 는 이러한 위협을 체계적으로 분류함

### 10.6.1 에이전트 고유 보안 위협

- 에이전트가 도구를 호출하고 외부 시스템에 영향을 미치므로
  - 단순한 텍스트 생성과는 차원이 다른 위험이 존재함

- **프롬프트 인젝션(Prompt Injection)**
  - 악의적 입력이 에이전트의 시스템 프롬프트를 우회하거나 덮어쓰는 공격
  - 종류
    - **직접 인젝션**: 사용자 입력에 지시문을 삽입
    - **간접 인젝션**: 검색된 문서나 도구 응답에 숨겨진 지시문
  - ⚠ 에이전트가 도구를 호출하는 능력을 갖추었기 때문에, 인젝션 성공 시 피해가 텍스트 생성보다 훨씬 큼

- **도구 오용(Tool Misuse)**
  - 에이전트가 의도하지 않은 방식으로 도구를 호출하는 것
  - → 예시: 파일 시스템 접근 도구가 민감한 경로를 읽거나, API 호출 도구가 인가되지 않은 엔드포인트에 요청을 보내는 경우

- **데이터 유출(Data Exfiltration)**
  - 에이전트가 접근한 내부 데이터를 외부로 전송하는 것
  - 방법: 도구 호출의 파라미터에 민감 정보를 포함시키거나, 외부 API 응답으로 데이터를 누출

- **과도한 자율성(Excessive Agency)**
  - 에이전트에게 필요 이상의 권한이 부여되어 의도하지 않은 범위의 작업을 수행하는 것
  - 원인: **최소 권한 원칙(Principle of Least Privilege)** 위반에서 비롯됨

## 10.7 다층 가드레일 아키텍처

- **가드레일(guardrail)** 이란
  - 에이전트의 동작을 안전한 범위 안에 유지하는 보호 장치
- 단일 계층이 아니라, 입력부터 출력까지 여러 단계에 걸친 **다층 방어**가 효과적

### 10.7.1 입력 가드레일

- 사용자 입력이 에이전트에 전달되기 전에 검사
- 포함 기능
  - **PII(개인식별정보, Personally Identifiable Information) 자동 제거**
  - **탈옥(jailbreak, 제한을 우회하려는 시도) 탐지**
  - **입력 길이 제한**
- 프롬프트 인젝션 탐지 모델을 배치하여 악의적 입력을 사전에 차단할 수 있음

### 10.7.2 실행 샌드박싱

- 에이전트가 코드를 실행하거나 외부 시스템에 접근할 때, 격리된 환경에서 수행
- 기술 수단
  - **gVisor**나 **Firecracker microVM**으로 실행 환경을 격리
  - 네트워크 이그레스 허용목록(allowlist, 외부로 나가는 네트워크 트래픽 허용 목록)으로 외부 접근 범위를 제한
- 명시적 정의 대상
  - 에이전트가 접근할 수 있는 파일 경로
  - 에이전트가 접근할 수 있는 API 엔드포인트
  - 에이전트가 접근할 수 있는 네트워크 범위

### 10.7.3 출력 가드레일

- 에이전트의 출력이 사용자에게 전달되기 전에 정책 규칙을 검사
- 포함 기능
  - 유해 콘텐츠 필터링
  - 민감 정보 마스킹(masking, 가림 처리)
  - 조직 정책 준수 여부 확인
- 이 계층은 환각 방지보다는 **정책 위반 방지**에 초점을 맞춤

### 10.7.4 ID/접근 제어

- 에이전트별로 **최소 권한 IAM(Identity and Access Management, 신원 및 접근 관리)** 을 적용
  - → 예시: 연구자 에이전트에게는 읽기 권한만 부여
  - → 예시: 실행 에이전트에게는 특정 도구 호출 권한만 부여
- 도구 호출 시마다 호출 주체의 권한을 확인하는 **런타임 검증**이 필요

### 10.7.5 프레임워크 내장 가드레일

- 주요 프레임워크는 가드레일을 내장 기능으로 제공
  - **OpenAI Agents SDK**
    - `guardrails` 프리미티브(기본 구성 요소)로 입력/출력 가드레일을 에이전트 정의에 포함 가능
  - **Google ADK**
    - safety 설정으로 유해 콘텐츠 카테고리별 차단 수준을 조정 가능
- 권장 방식
  - 프레임워크 내장 기능을 활용하되
  - 조직 정책에 맞는 커스텀 가드레일을 추가하는 것이 바람직함

## 10.8 에이전트 평가 프레임워크

- 에이전트 시스템은 단순한 텍스트 생성과 달리 복합적인 능력을 평가해야 함
  - 멀티스텝 계획(여러 단계에 걸친 계획 수립)
  - 도구 사용
  - 정책 준수
- 전용 벤치마크와 평가 방법론이 발전하고 있음

**표 10.2** 주요 에이전트 평가 벤치마크

| 벤치마크 | 평가 대상 | 핵심 특징 |
|---------|----------|----------|
| GAIA | 멀티스텝 계획 | 웹 검색·파일 처리를 결합한 실세계 질문 |
| WebArena | 웹 상호작용 | 실제 웹사이트에서 작업 수행 |
| τ-bench | 정책 준수 | 도메인 정책을 따르며 사용자와 대화 |
| Terminal-Bench | CLI 워크플로우 | 터미널 환경에서 작업 수행 |

### 10.8.1 CLEAR 프레임워크

- 에이전트를 5가지 차원에서 종합 평가하는 프레임워크
- 5가지 측정 차원
  - **C**ost(비용)
  - **L**atency(지연시간)
  - **E**fficacy(효과)
  - **A**ssurance(보증)
  - **R**eliability(신뢰성)
- 배경 관점
  - 정확도만으로는 프로덕션(실제 운영 환경) 적합성을 판단할 수 없음
  - 비용·지연시간·신뢰성을 함께 평가해야 함

### 10.8.2 LLM-as-Judge

- LLM을 평가자로 활용하는 접근
  - 스케일(대규모 평가)이 필요한 경우 빠르게 확산되고 있음
- 성능
  - LLM이 에이전트의 출력을 채점할 때, 사람 평가자와의 일치도가 약 **90%** 에 달한다는 연구 결과 있음
- ⚠ 한계
  - **위치 편향(Position Bias)**: 첫 번째 응답을 선호하는 경향
  - **장문 편향(Length Bias)**: 긴 답변을 선호하는 경향
  - **자기 편향(Self-Preference Bias)**: 같은 모델의 출력을 선호하는 경향
- 권장 방식: 사람 평가와 병행하는 것이 권장됨

### 핵심 구현

- 검증 워크플로우는 다섯 개의 노드로 구성

```python
graph.add_node("generate", generate_answer)
graph.add_node("consistency", check_consistency)
graph.add_node("grounding", check_grounding)
```

- 각 노드의 역할

  - **consistency 노드**: 3회 샘플링으로 일관성을 검사
  - **grounding 노드**: LLM을 활용하여 답변이 컨텍스트에 근거하는지 평가
  - **gate 노드**: 두 검증 결과를 종합하여 통과·재시도·에스컬레이션 중 하나를 결정

## 10.10 실패 사례 수집과 개선

- 검증 파이프라인은 문제를 탐지할 뿐 아니라
  - 탐지된 문제로부터 학습하는 **피드백 루프**를 구성해야 함
  - 실패 사례를 체계적으로 수집하고 분석하면 시스템을 지속적으로 개선할 수 있음

### 실패 사례 분류

- **유형**으로 분류
  - 일관성 실패
  - 출처 실패
  - 형식 오류
  - 기타
- **심각도**로 분류
  - 사소한 문제
  - 주요 문제
  - 치명적 문제
- 분류의 목적: 우선순위 결정과 자원 배분에 도움이 됨

### 패턴 분석

- 축적된 실패 사례에서 패턴을 분석
  - 특정 유형의 질문에서 일관성이 낮은 경우
    - 원인: 해당 도메인의 컨텍스트가 부족한 것일 수 있음
  - 특정 소스에서 출처 점수가 낮은 경우
    - 원인: 해당 문서의 품질이나 청킹(chunking, 문서를 작은 단위로 나누는 방법) 전략을 검토해야 함

### 피드백 루프

- 분석 결과는 세 가지 방향으로 피드백됨
  - **프롬프트 개선**: 가장 빠르게 적용할 수 있는 방법
  - **검색 전략 수정**: 청킹 크기, 검색 알고리즘, 리랭킹(reranking, 검색 결과 재순위 부여) 모델 등을 조정
  - **검증 기준 조정**: 임계값을 도메인에 맞게 튜닝(tuning, 세밀하게 조정)하는 것

## 10.11 실패 사례와 교훈

- 검증 파이프라인을 운영하면서 발생할 수 있는 문제와 대응 방법을 정리

### 과도한 검증으로 인한 지연

- 문제
  - 모든 답변에 3회 샘플링과 출처 검증을 수행하면 응답 시간이 크게 증가함
  - ⚠ 본 실습에서도 단일 질문에 11~17초가 소요됨
- 대응
  - 실시간 응답이 중요한 시스템에서는 검증 수준과 응답 속도 사이의 균형을 찾아야 함
- 대안적 접근: **위험도에 따라 검증 수준을 차등화**
  - 일반적인 질문: 가벼운 검증만 수행
  - 중요한 의사결정과 관련된 질문: 엄격한 검증을 적용

### False Positive와 False Negative

- ⚠ **False Positive(거짓 양성)** 문제
  - 키워드 기반 일관성 검사는 의미적으로 유사하지만 다른 단어를 사용한 답변을 실패로 처리할 수 있음
- ⚠ **False Negative(거짓 음성)** 문제
  - 임계값이 너무 낮으면 실제 문제 있는 답변이 통과할 수 있음
- 대응 방법
  - 의미론적 유사도(semantic similarity)를 활용하거나
  - 도메인별로 임계값을 조정해야 함

### 사람 승인 병목

- ⚠ 모든 실패가 사람에게 에스컬레이션되면 병목(bottleneck)이 발생함
- 완화 방법
  - 자동으로 처리할 수 있는 실패 유형을 늘리거나
  - 답변 불가 메시지를 반환하는 전략을 고려해야 함
